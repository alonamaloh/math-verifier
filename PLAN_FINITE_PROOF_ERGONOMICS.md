# Finite proof ergonomics

## Goal

Make small finite arguments read like their mathematical certificates:

```text
Matrix.Represents(F, 6)
    by Matrix.represents_by_witness(vector := ⟨2, 1, 1⟩)

∀ (n : ℕ). 2 ≤ n → n < 7 → P(n)
    by finite_check n from 2 until 7
```

Both forms must elaborate to ordinary kernel terms. No host-side Boolean result
is trusted, and downstream theorem statements retain the existing
`Matrix.Represents` and bounded-universal vocabulary.

## Design constraints

- Keep the kernel unchanged.
- Keep represented values and forms goal-inferred; the witness vector is the only
  information normally written at the call site.
- Make vector literals expected-type directed and reject length mismatches
  during elaboration.
- Give finite ranges half-open semantics: `from a until b` means
  `a ≤ n < b`.
- Enumerate only closed numeral endpoints in the first implementation.
- Put a configurable cap on the number of generated cases before any proof
  search begins.
- Close each case independently and report the exact numeral whose leaf did
  not close.
- The range checker may be expensive, but it must terminate predictably and
  must not hide a call to the unrestricted auto-prover over the entire
  universal goal.

## W1. RingVector tuples

Extend the existing anonymous tuple expression `⟨e₀, …, eₙ₋₁⟩` to recognize
an expected `RingVector(r, n)` type. RingVector is function-encoded rather than
inductive, so the tuple elaborator builds the ordinary library construction
from `RingVector.oneCoordinate` and `RingVector.appendCoordinate`. No new
surface or core expression is needed. The expected type supplies the
coefficient carrier to every coordinate.

Initial scope:

- nonempty literals;
- comma-separated expressions;
- square-bracket indexing remains the sole meaning of `[]` (`v[i]` is still
  the registered indexing operator);
- diagnostics distinguish an empty literal, a non-vector expected type, and
  an expected dimension different from the literal length.

Acceptance probes:

- an integer `RingVector(..., 3)` tuple elaborates and computes at all three
  indices;
- numerals and negative numerals inherit the expected Integer carrier;
- a length-two literal is rejected where dimension three is expected;
- existing postfix-index parsing and tests are unchanged.

## W2. Representation by witness

Add the generic theorem

```text
Matrix.represents_by_witness
    (A) (value) (witness)
    (computes : Matrix.quadraticForm(A, witness) = value)
    : Matrix.Represents(A, value)
```

The normal citation supplies only `vector := [...]`; goal unification recovers
`A` and `value`, and the auto-prover checks the closed quadratic-form equality.
If computation does not close, the theorem remains usable with an explicit
`computes := ...` argument.

Acceptance probe: the sum-of-three-squares form represents 6 by `⟨2, 1, 1⟩`
in one citation.

## F1. Kernel-visible finite-range certificate

Define an inductive proposition `Natural.AllFrom(P, start, count)`:

- the empty constructor certifies count 0;
- the prepend constructor stores `P(start)`, a certificate beginning at a
  canonical `next`, and an explicit proof `next = start + 1`.

The explicit successor equality is intentional. Integer addition is opaque,
so forcing the tail to be indexed by the expression `start + 1` would turn a
readable numeral table into an opaque addition chain. Recording the equality
lets every leaf retain its canonical numeral spelling while keeping the
whole step kernel-visible.

Prove once that an `AllFrom` certificate entails
`P(n)` whenever `start ≤ n < start+count`. Add the analogous Integer-facing
bridge, with a Natural count and successive Integer values.

Concrete certificates reduce to conjunctions of ground propositions. This is
the proof object generated by `finite_check`; it is not exposed in consumer
theorem statements.

## F2. `finite_check`

Add the proof form

```text
by finite_check n from LOWER until UPPER
```

It accepts a goal whose leading quantified variable is `n` and whose premises
provide the stated half-open bounds. The elaborator:

1. verifies that the carrier is Natural or Integer;
2. parses the closed endpoints and computes the case count;
3. rejects an empty/reversed range mismatch or a count above
   `MATH_FINITE_CHECK_MAX_CASES` (default 256);
4. constructs the corresponding `AllFrom` proposition;
5. proves each leaf separately using the ordinary local auto-prover with an
   independent effort budget;
6. applies the library bridge to produce the original universal statement.

The failure diagnostic names the first unproved case, for example:

```text
finite_check: case n = 4 did not close
```

Nested checks are allowed and multiply their counts normally; the cap is
checked at each invocation.

Acceptance probes:

- Natural half-open range with nonzero lower bound;
- Integer range crossing zero;
- a false leaf reports its exact value;
- a range over the configured cap declines before proof search;
- a symbolic endpoint declines with an actionable message;
- the generated declaration contains an ordinary Pi/implication statement,
  not `AllFrom`.

## F3. Apply to rank-three truants

Use witness literals for the positive tables. For negative statements, prefer
the smallest range sweep after mathematical bounds or residual reductions;
do not blindly enumerate three coordinates when one residual coordinate
suffices.

Target representatives over `x²+2y²`:

- diagonal corners 1, 2, 3, 4, 5, with expected truants 14, 7, 10, 14, 10;
- the already-known sum of three squares, truant 7;
- odd-border corners 4 and 5, both expected truant 7.

The final gate is the full library, ordinary tests, error tests, elaborated
statement checks, and clean-style ratchet with no budget increase.
